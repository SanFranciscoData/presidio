Task Prompt Quality Spec — requirements a generated task prompt (the public instruction an AI coding agent sees) must satisfy. A prompt should read like a human handing a coding agent real work — not a benchmark task narrating itself.

Length & format: ≤ 2000 characters (hard ceiling; rewrite/compress to fit, never truncate away requirements). Aim ~300 words or fewer (advisory). No Markdown section headers (#/##/### or setext). Avoid bulleted lists — prefer prose; a short bullet list of concrete literals is OK only when it aids clarity. No boilerplate template labels as structure (Objective:, Requirements:, Deliverables:, Steps:, Acceptance Criteria:, Verification:). Not "obviously LLM-synthesized" — no padding/restated context/redundant enumeration.

Voice & framing: No persona self-introduction opener (e.g. "Hi! I'm a biophysics researcher…"); open with the work. No conversational sign-off (e.g. "Let me know when you're done!"). No suggestive/leading language: no role-play flattery ("you are an expert in X"), no process nudges ("think step by step", "carefully", "make sure to"). Written in the requester's own voice, as a task spec.

Fourth wall / evaluation leakage: Never tell the agent it is being tested, graded, benchmarked, or evaluated. Never describe the verifier/test harness/grading mechanism (forbidden: "we will verify…", "the verifier will…", "your solution will be evaluated by…", "a randomized fuzz-testing suite", "we will capture/relocate/temporarily move…", "anti-delegation checks", "over N random inputs", "byte-exact match of the oracle's output"). Instead state the user-facing contract the deliverable must satisfy. Example transform: replace a described relocation/fuzz harness with "your program must independently reproduce the reference's exact output for any input; it must not invoke, exec, import, symlink, or copy /app/ref."

Environment / infrastructure leakage: No harness/environment narration ("pre-installed in the environment", "we provide X"). Tooling constraints phrased as requirements ("use only the Python standard library and numpy"), not assumptions about a preconfigured container.

Benchmark scaffolding: No canary markers, GUIDs, or harness comments (e.g. <!-- harbor-canary GUID <uuid> -->, <!-- canary: … -->).

Paths & concreteness: Absolute paths only (/app/…, /home/user/…); never relative (./data.csv, ../cfg, dir/file.ext). Preserve all concrete requirements — no rewrite may drop a load-bearing literal: absolute paths, file names, ports, formats, numeric values/thresholds, the entry point, or any must/must-not rule. Specify exact output contracts precisely enough for an automated test.

Outcome orientation: Describe WHAT, not HOW — goal + requirements + exact output contract; do not narrate solution steps. No solution walkthrough (no "first do X then Y", no suggested algorithms, no solution code).

Scenario realism (flags): Industrial-grade real-world scenario, not a demo/toy/algorithmic-competition (LeetCode/math-competition) problem. Not exam/competition-style.
